# RAG 毕设项目梳理文档

## 1. 项目概述

这是一个基于 **RAG（Retrieval-Augmented Generation，检索增强生成）** 的智能问答系统，核心目标是让用户先创建知识库、上传文档，再针对知识库内容进行提问，系统通过“文档检索 + 大模型生成”的方式返回答案。

项目整体分为两部分：

- 前端：`front_end`，基于 `Vue 3 + Vite + Element Plus`
- 后端：`rear_end`，基于 `Django + Django REST Framework + MySQL + FAISS`

系统同时包含：

- 普通用户端
- 管理员后台
- 订阅与支付模块
- 历史对话模块

---

## 2. 项目定位

从功能上看，这个项目更像一个“私有知识库智能问答平台”，适合以下场景：

- 用户上传课程资料、论文、规章制度、笔记
- 系统将文档切分并向量化
- 用户围绕自己的文档进行问答
- 管理员可统一管理平台用户、知识库和文档

如果你在答辩中需要一句话介绍，可以直接这样说：

> 本项目是一个面向私有资料场景的 RAG 智能问答平台，用户可构建个人知识库并基于上传文档进行智能问答，系统支持订阅套餐、历史对话和管理员全局管理。

## 5. 系统业务流程

### 5.1 普通用户流程

1. 用户注册并登录系统
2. 创建自己的知识库
3. 上传 `txt / md / pdf` 文档
4. 后端对文档进行解析、切分、向量化并写入 FAISS
5. 用户进入问答页面输入问题
6. 后端从知识库中检索相关片段
7. 将检索结果与用户问题一起发送给大模型
8. 返回生成答案并记录历史问答

### 5.2 管理员流程

1. 管理员登录后台
2. 查看全站用户列表
3. 管理全站知识库
4. 查看指定知识库下文档
5. 删除违规文档或异常知识库
6. 查看平台统计数据

### 5.3 订阅流程

1. 用户进入订阅页查看套餐
2. 发起支付宝支付
3. 支付回调更新订单状态
4. 系统开通对应订阅权限
5. 用户每日问答次数与知识库数量限制随套餐变化

## 6. 界面功能说明

以下内容以项目当前真实路由和页面实现为准。

### 6.1 登录页 `/login`

- 用户输入用户名和密码登录
- 登录成功后保存 `access token` 和 `refresh token`
- 自动拉取当前用户信息
- 支持根据跳转参数返回原目标页

### 6.2 注册页 `/register`

- 用户输入用户名、密码、邮箱进行注册
- 注册成功后跳转登录页

### 6.3 知识库列表页 `/kb`

- 展示当前用户拥有的知识库
- 支持创建知识库
- 支持删除知识库
- 支持进入“文档管理”页面
- 支持进入“智能问答”页面

### 6.4 文档管理页 `/kb/:kbId/documents`

- 查看某个知识库下的文档列表
- 上传文档到知识库
- 支持文件类型：`txt / md / pdf`
- 支持重名策略：
  - `keep`：保留原文件并使用随机名
  - `replace`：替换原文件
- 支持删除文档
- 支持预览文档内容

### 6.5 智能问答页 `/kb/:kbId/chat`

- 基于指定知识库进行问答
- 展示今天剩余问答次数
- 支持流式输出答案
- 支持多轮会话，带 `session_id`
- 显示回答耗时
- 显示 Token 使用情况
- 支持 Markdown 渲染回答

### 6.6 个人信息页 `/profile`

- 查看当前账号信息
- 修改用户名和邮箱
- 修改密码
- 跳转到订阅页面

### 6.7 订阅页 `/subscription`

- 查看套餐列表
- 查看当前订阅状态
- 查看剩余问答额度
- 发起支付宝支付
- 支付成功后刷新订阅信息

### 6.8 历史对话页 `/chat-history`

- 查看历史问答列表
- 按时间查看历史记录
- 查看单条对话详情
- 删除历史记录

### 6.9 管理员用户管理页 `/admin/users`

- 查看全站用户
- 搜索用户
- 新建用户
- 修改用户信息
- 重置用户密码
- 删除用户
- 设置是否为管理员

### 6.10 管理员知识库页 `/admin/kbs`

- 查看全站知识库
- 按用户 ID 过滤
- 为指定用户创建知识库
- 修改知识库信息
- 删除知识库
- 查看知识库下文档
- 删除文档

### 6.11 管理员统计页 `/admin/stats`

- 展示全站用户数
- 展示知识库数量
- 展示文档数量
- 展示聊天数量
- 展示订阅统计
- 展示今日使用情况

说明：

- 后端已经实现 `GET /api/admin/stats`
- 但前端当前页面仍然使用模拟数据，不是实时接口数据

## 9. 核心 API 详细说明

这里挑毕设里最核心、最适合讲解的接口说明。

### 9.1 创建知识库

- 方法：`POST`
- 路径：`/api/knowledge/create`
- 功能：为当前登录用户创建一个新的知识库

请求体示例：

```json
{
  "name": "数据库课程资料",
  "description": "用于存放数据库原理相关文档"
}
```

返回示例：

```json
{
  "id": 1,
  "name": "数据库课程资料",
  "description": "用于存放数据库原理相关文档",
  "faiss_path": "E:\\code\\RAG\\rear_end\\faiss_indexes\\user_1\\kb_1.index",
  "created_at": "2026-02-20T12:34:56.123Z"
}
```

### 9.2 上传文档入库

- 方法：`POST`
- 路径：`/api/knowledge/upload`
- 功能：上传文档并完成切分、向量化、写入索引
- 请求类型：`multipart/form-data`

表单字段：

| 字段 | 说明 |
| --- | --- |
| `kb_id` | 知识库 ID |
| `file` | 上传文件 |
| `on_conflict` | 重名策略，`keep` 或 `replace` |

返回示例：

```json
{
  "id": 3,
  "kb_id": 1,
  "filename": "操作系统复习提纲.pdf",
  "chunk_count": 28,
  "uploaded_at": "2026-02-20T12:34:56.123Z"
}
```

### 9.3 智能问答

- 方法：`POST`
- 路径：`/api/rag/chat`
- 功能：根据知识库内容生成回答

请求体示例：

```json
{
  "kb_id": 1,
  "question": "什么是事务的 ACID 特性？",
  "session_id": "可选，会话ID"
}
```

返回示例：

```json
{
  "answer": "事务的 ACID 特性包括原子性、一致性、隔离性和持久性。",
  "elapsed_ms": 1480,
  "token_usage": {
    "prompt_tokens": 286,
    "completion_tokens": 97,
    "total_tokens": 383
  },
  "session_id": "0ef2e91a-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
}
```

### 9.4 获取历史对话

- 方法：`GET`
- 路径：`/api/rag/history`
- 功能：分页获取历史问答记录

常用查询参数：

| 参数 | 说明 |
| --- | --- |
| `kb_id` | 可选，按知识库过滤 |
| `page` | 页码 |
| `page_size` | 每页数量 |

### 9.5 支付下单

- 方法：`POST`
- 路径：`/api/payment/alipay`
- 功能：创建订单并返回支付链接

请求体示例：

```json
{
  "plan_id": 2
}
```

返回示例：

```json
{
  "pay_url": "https://openapi.alipaydev.com/gateway.do?...",
  "order_id": "8a5d9c5c2d2d4f8b"
}
```

## 10. 数据与模块关系

- 用户 `User`
- 订阅计划 `SubscriptionPlan`
- 用户订阅 `UserSubscription`

- 文档切片 `DocumentChunk`
- 聊天历史 `ChatHistory`

关系可以这样理解：

- 一个用户可以拥有多个知识库
- 一个知识库可以有多个文档
- 一个文档会被切分成多个文档块
- 用户每次问答会生成一条历史记录
- 用户可以购买某个订阅套餐

## 13. 当前发现的问题

这是我在阅读代码后发现的几个比较重要的问题，你后面完善项目时可以优先处理。

### 13.1 旧 API 文档与真实代码不一致

`rear_end/API.md` 中很多接口写成了：

- `/api/kb/...`
- `/api/document/...`

但当前真实后端代码使用的是：

- `/api/knowledge/...`
- `/api/knowledge/document/...`

因此：

- 旧文档不能完全作为最终答辩依据
- 正式接口说明建议以 Django 路由代码为准

### 13.2 文档别名路由存在冲突风险

项目里额外挂载了 `knowledge/document_urls.py`，其中部分路径与知识库删除路径过于接近，存在路由冲突和遮蔽风险。

简单说就是：

- 某些“文档删除别名接口”可能会被“知识库删除接口”优先匹配

这个问题建议后续统一接口命名，避免使用歧义路径。

### 13.3 管理员统计页前后端未真正打通

- 后端已经实现 `/api/admin/stats`
- 前端 `AdminStatsView.vue` 还在使用模拟数据

所以目前这个页面更像是“静态展示页”，还没有完全接入真实统计。

### 13.4 存在一个损坏的前端页面文件

`front_end/src/views/ProfileView.vue` 文件内容已经损坏，但当前路由实际使用的是 `UserProfileView.vue`，所以系统暂时还能运行。

建议：

- 删除废弃损坏文件
- 或修复后统一保留一个个人信息页面

### 13.5 后端目前是开发态配置

从设置上看：

- `DEBUG = True`
- `ALLOWED_HOSTS = ['*']`
- `SECRET_KEY` 直接写在配置里

这适合本地开发，但不适合正式部署。

## 15. 答辩时可以怎么讲

如果老师问“这个系统是怎么工作的”，你可以按下面顺序回答：

1. 前端负责用户交互，包括登录、知识库管理、文档上传和问答页面
2. 后端使用 Django 提供 REST API，负责用户、知识库、问答、支付等业务
3. 用户上传文档后，系统会解析文本并切分成小块
4. 系统调用向量模型把文本块转成向量，并存入 FAISS 索引
5. 用户提问时，系统先在知识库中检索最相关的文本片段
6. 再把这些片段与问题拼接成 Prompt 发给大模型
7. 最后返回答案，并保存历史记录与 Token 使用信息

如果老师问“你的工作量体现在哪”，你可以重点讲：

- 前后端完整联调
- 知识库与文档管理
- RAG 检索问答链路
- 流式输出与历史对话
- 订阅与支付
- 管理员平台
